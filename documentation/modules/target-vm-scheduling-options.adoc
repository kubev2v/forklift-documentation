= Target VM scheduling options

You can use the following options to schedule when your target VMs are powered on:

* *Node Selector* rule: This is both the simplest and strictest rule. You define a set of mandatory exact match key/value label pairs that the target node must possess. If no node on the cluster has all the labels specified, the VM is not scheduled and it remains in a `Pending` state until there is space on such a node.

For example, you might have two service levels, gold (premium) and silver (regular), and you want to migrate VMs for the premium service to a specific node. You could create a key named `license-tier` and a value named `gold`. You could then use the *Node Selector* option to create a rule that certain VMs can only be switched `on` if migrated to a node for which `license-tier=gold` is true. Even if there is space on a node for which `license-tier=silver` is true, your VMs will status `Pending`, until there is space on a node that meets your condition. 

* *Node Affinity and Node Anti-Affinity* rules: These rules offer more flexible placement control than rigid Node Selector rules. _Node Affinity_ rules let you schedule VMs to run on selected nodes. _Node Anti-affinity_ rules let you prevent VMs from being scheduled to run on selected nodes. 

Both types of rules allow you to include both _hard_ and _soft_ conditions in the same rule. A hard condition is a requirement, and a soft condition is a preference.

To understand the difference between a Node Selector rule and an Node Affinity rule with a hard rule, consider these examples: 

* A Node Selector rule that requires a VM must be migrated only to a specific cluster
* A Node Affinity rule with a hard condition that requires a VM must be migrated to a specific cluster _or_ that it be migrated to an SSD disk, _but_ it cannot be migrated to node for which `license-tier=gold` is true. 

** Node Affinity rules 


Node Affinity: Defines rules to require (hard rule) or prefer (soft rule) that the VM is scheduled on nodes that match certain labels.

Node Anti-Affinity: Defines rules to require (hard rule) or prefer that the VM avoids scheduling on nodes that match certain characteristics. This is vital for High Availability (HA), as it ensures redundant VMs are separated across different failure domains (e.g., nodes in separate availability zones).

Custom Scheduler Name: If your OpenShift environment uses a secondary or specialized scheduler (in addition to the default kube-scheduler) to handle specific workload types, you can instruct MTV to apply this custom scheduler name to the target VM's manifest. This directs the VM to use the specialized logic designed for that workload.

By integrating these controls into the migration plan, you ensure that the complex scheduling logic required for modern applications is defined upfront, preventing post-migration performance degradation or unexpected scheduling errors.
