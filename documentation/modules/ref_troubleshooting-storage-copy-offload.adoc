= Troubleshooting storage copy offload

[role="_abstract"]
Resolve these problems that are unique to storage copy offload.

== Ambiguous storage mapping errors

In previous versions, migrations could fail with ambiguous storage mapping errors when multiple storage arrays from the same vendor were configured in the storage map. {project-short} now automatically resolves these ambiguities by matching the Network Address Authority (NAA) identifier of each Raw Device Mapping (RDM) disk with the backing device NAA identifiers of candidate datastores.

If you encounter storage mapping errors, verify the following:

* The storage arrays are properly configured and accessible.
* The NAA identifiers are correctly reported by the storage arrays.
* If using custom vendor mappings, the `forklift-naa-oui-map` ConfigMap is correctly configured.

For more information about multi-array support and NAA-based disambiguation, see xref:con_multi-array-rdm-support_{context}[Multi-array Raw Device Mapping support for storage copy offload].

== Custom NAA-to-vendor mapping issues

If you have configured custom NAA-to-vendor mappings by using the `forklift-naa-oui-map` ConfigMap and the mappings are not working as expected:

. Verify that the ConfigMap is created in the correct namespace.

. Check that the NAA or OUI prefix format is correct in the ConfigMap.

. Verify that the ConfigMap changes have been picked up by the controller.

. Ensure that there are no syntax errors in the ConfigMap YAML.

For more information about configuring custom mappings, see xref:proc_configuring-naa-oui-map_{context}[Configuring custom NAA-to-vendor mappings].
