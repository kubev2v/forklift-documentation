= About Deep Inspection

[role="_abstract"]
You can use Deep Inspection to analyze virtual machine (VM) disk images before migration. This finds compatibility issues that standard inspection might miss and helps you plan remediation before starting the migration.

[IMPORTANT]
====
Deep Inspection is a Technology Preview feature only. Technology Preview features
are not supported with Red Hat production service level agreements (SLAs) and
might not be functionally complete. Red Hat does not recommend using them
in production. These features provide early access to upcoming product
features, enabling customers to test functionality and provide feedback during
the development process.
====

Prerequisites::

* {vmw} vSphere source provider configured in {project-short}.
* {vmw} Virtual Disk Development Kit (VDDK) configured for the source provider. Deep Inspection uses VDDK to access VM disk images.
* Changed Block Tracking (CBT) must be enabled.
* Cluster resources available to run inspection pods. Each pod needs CPU, memory, and storage to mount and analyze VM disks.

Limitations::

* Deep Inspection is available only for {vmw} vSphere providers. It is not available for other source providers.
* {project-short} limits concurrent inspections to 10 by default. Additional inspections are queued until resources become available.

Considerations::

* Deep Inspection takes longer than standard inspection. It creates a VM snapshot and inspects the base disks, then performs detailed analysis.
* Inspection time depends on VM disk size and cluster resources.

== Inspection methods

{project-short} supports two methods for inspecting VMs before migration:

* Standard inspection: Queries the source provider's API to collect VM metadata. This includes operating system information, disk configuration, and hardware settings. This method is fast. It does not need extra resources. However, it relies on metadata from the provider.

* Deep Inspection: Launches a temporary worker pod in the {ocp-short} cluster. The pod mounts and analyzes the VM's disk images directly. This method gives you more confidence in migration success. It finds guest-level issues that standard inspection might miss.

== How Deep Inspection works

Deep Inspection launches a temporary pod in your {ocp-short} cluster that uses VDDK to mount the VM's disk images. {project-short} creates a snapshot to analyze the guest operating system without affecting the VM's power state, whether it is stopped or running. After analysis, {project-short} cleans up the pod and stores the results.

Deep Inspection provides detailed information about the VM to help assess migration readiness:

[cols="30%,70%",options="header"]
|===
|Category
|Information analyzed

|Operating system and hardware
a|
* Operating system name, version, and architecture
* kernel version

|Storage and file systems
a|
* File system types (for example, ext4, xfs, ntfs)
* Partition layouts and mount points
* Root device path (critical for dual-boot or multi-disk VMs)
|===

== When to use Deep Inspection

Use Deep Inspection for:

* Critical or production VMs where you need high confidence in successful migration.
* Legacy VMs where driver and operating system support is uncertain.
* Dual-boot or multi-disk VMs that need the correct root device identified.
* VMs with standard inspection warnings that need more detailed information.
